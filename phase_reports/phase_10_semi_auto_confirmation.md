# Phase 10 — SemiAuto 执行前确认

## 已完成内容

### 1. 半自动模式已改为“先确认，再执行”
已更新：

- `aibot/Scripts/Agent/Handlers/SemiAutoModeHandler.cs`

本阶段继续严格对齐 `claude_plan.md` 中：

- 第 19 条：自然语言意图 → Skill
- 第 21 条：半自动模式可显示“确认执行”按钮
- 第 32.G.32：加入执行前确认

此前 `SemiAutoModeHandler` 的行为是：

1. 解析玩家输入
2. 找到 Skill
3. 立即执行

这与 `plan` 中“可执行或先给出确认提示后执行”的目标仍有缺口。

本阶段已改为：

1. 解析玩家输入
2. 命中 Skill 后先检查 `CanExecute()`
3. 若可执行，则生成待确认动作摘要
4. 不立刻执行，只进入 pending 状态
5. 玩家确认后才真正调用 `ExecuteAsync()`

这样半自动模式终于具备了更安全、更符合产品定义的交互闭环。

---

### 2. 已新增待确认动作状态管理
当前 `SemiAutoModeHandler` 内部已新增待确认执行状态，用于保存：

- 待执行的 Skill
- 对应的解析结果与参数
- 原始输入
- 提示文案

这让半自动模式可以明确区分两类阶段：

- “正在解析新的用户意图”
- “正在等待玩家确认一个已识别的动作”

同时避免了在玩家还未确认前就提前改变游戏状态。

---

### 3. 已支持文本确认 / 取消指令
当前 `SemiAuto` 已支持以下确认类文本输入：

- `确认执行`
- `确认`
- `yes`
- `y`

以及取消类输入：

- `取消执行`
- `取消`
- `no`
- `n`

因此即使 UI 按钮不可见或玩家更习惯键盘输入，仍然可以完成完整确认流程。

### 4. 聊天窗口已加入确认执行面板
已更新：

- `aibot/Scripts/Ui/AgentChatDialog.cs`

本阶段为 `SemiAuto` 聊天窗口新增了一个轻量的 pending 区域，包含：

- 待执行动作摘要文本
- `确认执行` 按钮
- `取消` 按钮

当半自动模式识别到一个可执行 Skill 时：

- 聊天历史中会输出待确认提示
- 底部 pending 面板会显示对应摘要
- 玩家可直接点击按钮完成确认或取消

这正好补上了 `plan` 中对半自动模式的交互建议。

### 5. 待确认提示会展示关键参数摘要
为了避免玩家只看到一个抽象技能名，本阶段增加了参数摘要展示。

例如待确认提示现在会尽量显示：

- 卡牌名
- 目标名
- 药水名
- 地图列/行
- 选项 ID
- 商品名
- Bundle 索引
- Crystal Sphere 坐标 / 占卜类型

这样玩家在确认前能更清楚知道 Agent 打算做什么，而不是“黑箱式执行”。

---

### 6. 取消时会正确清理 pending 状态
当前如果玩家选择取消：

- handler 内的 pending 执行状态会清空
- 聊天窗口的待确认面板会隐藏
- 会在决策日志中写入 `SkillCanceled`

如果玩家确认：

- pending 状态会先被消费并清空
- 然后才真正执行 Skill
- 决策日志中会写入正常执行或失败结果

这保证了半自动模式不会出现旧操作残留、重复确认或误执行的问题。

## 遇到的问题

### 问题 1：当前半自动执行链路过于直接
原有实现一旦命中 Skill 且 `CanExecute()` 为真，就会立刻执行，这与“半自动”应具备的玩家确认语义不一致。

### 解决方案
引入一个最小化的 pending 执行模型：

- 识别动作时只生成待确认对象
- 确认后再执行
- 取消时直接清空

这样无需大幅改动 Skill 抽象，就能补齐交互安全层。

---

### 问题 2：确认交互不能只依赖按钮
如果确认流程只放在按钮里，会导致：

- 键盘用户不便
- 某些 UI 显示异常时无法继续
- 后续自动测试 / 调试不方便

### 解决方案
同时支持：

- 按钮确认 / 取消
- 文本确认 / 取消命令

让 UI 与文本命令形成双通道兜底。

---

### 问题 3：待确认动作需要让玩家看得懂
如果只显示 Skill 名称，玩家并不能确定 Agent 到底准备操作哪张牌、哪瓶药水、哪个地图节点。

### 解决方案
在 pending 提示中补充关键参数摘要，让确认动作更可读、更可控。
