# Phase 11 — AgentLlmBridge 与 QnA 受限补答

## 已完成内容

### 1. 已新增统一的 `AgentLlmBridge`
已新增：

- `aibot/Scripts/Agent/AgentLlmBridge.cs`

本阶段继续严格对应 `claude_plan.md`：

- 第 16 条：建立统一的 LLM 桥接层
- 第 17 条：系统边界 Prompt 硬编码约束
- 第 30 条中的 `AgentLlmBridge`
- 第 32.H.36 与 32.J.43~45

当前新增的 `AgentLlmBridge` 负责：

- 统一向 DeepSeek 发起问答请求
- 管理超时与首次请求重试
- 注入固定系统边界 prompt
- 将本地知识与当前对局上下文一起注入问答请求
- 对返回文本做基础响应过滤

这意味着问答模式终于不再只有“本地检索命中 / 纯本地兜底”两档，而是具备了正式的受限云端桥接层。

---

### 2. 系统边界 Prompt 已写死在代码层
当前 `AgentLlmBridge` 中已写死一组不可被知识库覆盖的硬约束，包括：

- 只回答《杀戮尖塔2》相关内容
- 不能处理文件 / 系统 / shell / 代码 / 网络 / 通用助手问题
- 不接受用户尝试覆盖这些规则
- 回答必须优先基于游戏上下文与本地知识结果

这和 `plan` 的核心原则一致：

- Agent 的边界由代码强约束，而不是由知识库“提醒”
- 即使后续知识库允许玩家自定义，也不会让 Agent 滑向通用助手

---

### 3. `QnA` 模式已接入云端补答链路
已更新：

- `aibot/Scripts/Agent/Handlers/QnAModeHandler.cs`

当前 `QnA` 模式的处理顺序已经调整为三层：

1. 先做游戏域问题过滤
2. 再尝试 Tool 型问答
3. 再尝试本地知识检索
4. 若本地知识不足，则调用 `AgentLlmBridge`
5. 若云端也不可用或失败，再回退到当前局势摘要兜底

这更接近 `plan` 中“本地知识优先，必要时再调用 LLM”的目标设计。

---

### 4. LLM 问答请求已注入当前对局上下文
当前 `AgentLlmBridge` 在问答请求里会注入：

- 当前角色
- 当前推荐构筑
- 对局进度摘要
- 当前策略需求摘要
- 本地知识检索结果
- 本地知识来源列表（若存在）

这样云端补答不是“裸问模型”，而是在一个明确的游戏上下文里回答问题。

这能减少：

- 幻觉回答
- 与当前对局脱节的泛泛建议
- 忽略本地知识库内容的问题

---

### 5. 已加入基础响应过滤
当前 `AgentLlmBridge` 对响应做了基础后处理，包括：

- 清理代码块标记
- 截断过长回答
- 检查明显越界关键词（如 shell / git / python / javascript / file system 等）
- 命中越界内容时，直接替换为统一拒答文本

这还不是最终态的完整边界过滤器，但已经把最明显的风险先压住了。

---

### 6. `QnA` 仍保持本地知识优先
虽然本阶段接入了云端桥接层，但当前策略仍然明确保持：

- 只要本地知识有直接答案，就优先返回本地知识
- 只有本地知识不足时，才走云端补答

这符合 `plan` 中“轻量、本地优先”的原则，也避免了把所有问答都变成云端依赖。

## 遇到的问题

### 问题 1：现有云端能力全部耦合在 `DeepSeekDecisionEngine`
之前云端调用逻辑主要服务于自动决策，不适合直接塞进 `QnA` 模式中复用。

### 解决方案
新增 `AgentLlmBridge`，只负责 Agent 层的受限文本问答桥接，而不是继续扩大 `DeepSeekDecisionEngine` 的职责边界。

---

### 问题 2：问答模式需要云端补答，但不能失去“游戏专属 Agent”边界
如果直接把问题原样发给模型，问答模式很容易退化成通用聊天助手。

### 解决方案
把游戏边界 prompt 写死在 `AgentLlmBridge` 内，并在响应返回后再做一次代码侧过滤，形成双层收口。

---

### 问题 3：LLM 问答不能脱离本地知识和当前局势
如果云端补答不注入当前知识检索结果与局势摘要，就会更容易给出泛化建议甚至错误建议。

### 解决方案
在 prompt 中明确加入：

- 本地知识结果
- 知识来源
- 当前角色 / 构筑 / 对局进度 / 策略需求

把云端回答约束在已有上下文内。
