# Phase 07 — Assist 推荐层场景扩展

## 已完成内容

### 1. Assist 推荐层已扩展到更多关键场景
已更新：

- `aibot/Scripts/Ui/AgentRecommendOverlay.cs`

在第六阶段的基础上，本阶段继续扩展了 `Assist` 模式的推荐覆盖场景。新增支持：

- 战斗手牌推荐
- 商店商品推荐
- 休息点选项推荐
- 事件选项推荐

这样一来，辅助模式当前已覆盖：

- 卡牌奖励推荐
- 遗物选择推荐
- 地图节点推荐
- 战斗手牌推荐
- 商店商品推荐
- 休息点选项推荐
- 事件选项推荐

这让 `Assist` 模式从“局部可用”进一步提升为“贯穿更多关键决策点的推荐模式”。

---

### 2. 战斗手牌推荐已接入现有战斗决策逻辑
本阶段没有为战斗推荐单独实现一套评分系统，而是直接复用了现有：

- `DecisionEngine.ChooseCombatActionAsync()`

当前实现流程：

1. 在战斗出牌阶段读取当前玩家手牌
2. 过滤出可打出的牌
3. 收集当前可攻击敌人
4. 调用战斗决策接口
5. 将“推荐”标签贴到被选中的手牌 UI 上

这样可以保证：

- Assist 模式的战斗推荐和全自动模式共享同一套核心策略
- 推荐结果更一致
- 后续维护只需要继续增强原有决策引擎

---

### 3. 商店推荐已接入现有购买决策逻辑
当前商店推荐会：

1. 获取 `NMerchantRoom` 与打开的 `NMerchantInventory`
2. 收集当前可购买、且满足金币/药水槽条件的 `MerchantEntry`
3. 调用 `DecisionEngine.ChooseShopPurchaseAsync()`
4. 将推荐标签贴到对应 `NMerchantSlot` 上

这样能够直接把已有商店购买建议可视化，而不执行任何购买行为。

---

### 4. 休息点与事件推荐已接入
#### 休息点
当前实现会：

- 获取 `NRestSiteRoom`
- 收集可点击的 `NRestSiteButton`
- 调用 `DecisionEngine.ChooseRestSiteOptionAsync()`
- 将推荐标签贴到推荐按钮上

#### 事件
当前实现会：

- 获取 `NEventRoom`
- 收集可点击的 `NEventOptionButton`
- 调用 `DecisionEngine.ChooseEventOptionAsync()`
- 将推荐标签贴到推荐按钮上

这让辅助模式在战斗外的流程决策支持更完整。

---

### 5. 推荐层签名去重已同步扩展
为了避免在 `_Process()` 中频繁重复计算，本阶段同步扩展了推荐层的“场景签名”机制，当前会针对以下场景生成签名：

- overlay 卡牌奖励
- overlay 遗物选择
- 地图路线
- 战斗手牌
- 商店槽位
- 休息点选项
- 事件选项

只要当前界面结构没有变化，就不会重复刷新推荐，大幅降低了无意义计算频率。

## 未完成内容

### 1. 仍未覆盖的 Assist 场景
虽然辅助模式覆盖面已经明显提升，但以下场景仍未接入：

- 奖励领取推荐
- Bundle 选择推荐
- Crystal Sphere 推荐
- 宝箱特殊界面推荐
- 更细粒度的药水使用提示

### 2. 目前仍然是单标签策略
当前每次只保留一个推荐标签，不会同时对多个对象给出排序式提示。因此：

- 看不到次优选项
- 无法比较多个近似推荐
- 也没有“谨慎 / 高风险 / 高收益”之类的视觉区分

### 3. 推荐层仍为轻量实现
目前的推荐层依然保持轻量：

- 只有一个小型 `推荐` 标签
- 理由在 tooltip 中
- 没有更复杂的卡片式解释 UI

这符合当前阶段目标，但后续仍可继续增强。

## 遇到的问题

### 问题 1：战斗手牌推荐缺少显式的“手牌容器”入口
当前代码里并没有一个专门暴露给 Mod 的“手牌 UI 根节点”查询封装，因此很难直接精准拿到手牌区域控件。

### 解决方案
本阶段采用兼容策略：

- 全局查找可见 `NCardHolder`
- 再根据当前手牌中的 `CardModel` 做匹配
- 只对决策引擎返回的目标卡牌贴标签

这种方式不完美，但在当前结构下能稳定落地。

---

### 问题 2：商店推荐需要把 `MerchantEntry` 映射回 UI 槽位
决策引擎返回的是 `MerchantEntry`，而推荐层需要的是对应 `NMerchantSlot` 控件。

### 解决方案
本阶段直接复用 `NMerchantInventory.GetAllSlots()`，通过：

- `slot.Entry == decision.Entry`

来完成数据对象到 UI 节点的映射。

---

### 问题 3：推荐层支持更多场景后，重复刷新风险上升
如果不扩展签名机制，每次 `_Process()` 都可能重复计算推荐，带来额外开销和标签闪烁。

### 解决方案
继续沿用并扩展签名去重：

- 各场景独立生成签名
- 内容不变则不刷新
- 保持推荐层响应及时但不过度抖动

---
