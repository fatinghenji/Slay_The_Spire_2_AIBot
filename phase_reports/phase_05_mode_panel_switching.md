# Phase 05 — 模式切换面板与二次确认

## 已完成内容

### 1. 模式切换面板已落地
已新增：

- `aibot/Scripts/Ui/AgentModePanel.cs`

本阶段新增了一个独立的 `AgentModePanel`，用于统一展示和切换 Agent 的四种模式：

- `Full Auto`
- `Semi Auto`
- `Assist`
- `QnA`

当前面板具备以下能力：

- 显示当前模式
- 显示状态提示
- 提供四种模式按钮
- 当前模式按钮自动禁用
- 支持热键显示/隐藏（默认 `F8`）
- 使用独立 `CanvasLayer`，不依赖 `sts2/` 侧修改

这意味着从本阶段开始，四模式切换已经拥有了明确、统一的 UI 入口。

---

### 2. 二次确认流程已接入
本阶段没有另起一套切换逻辑，而是直接复用了 `AgentCore` 中已经存在的：

- `ModeChangeRequested`
- `ModeChanged`

当前实现流程：

1. 玩家点击模式按钮
2. `AgentCore.SwitchModeAsync()` 检查是否需要确认
3. 若需要确认，则触发 `ModeChangeRequested`
4. `AgentModePanel` 显示确认区
5. 玩家点击确认后，面板使用 `force: true` 再次发起切换
6. 切换成功后，`ModeChanged` 刷新当前模式显示

这样做的好处是：

- 模式切换逻辑仍然集中在 `AgentCore`
- UI 只负责交互与确认
- 不会导致状态机分裂到 UI 层

---

### 3. `AgentCore.Initialize()` 已接入模式面板创建
已更新：

- `aibot/Scripts/Agent/AgentCore.cs`

本阶段在初始化时增加：

- `AgentModePanel.EnsureCreated(runtime)`

这样模式面板和聊天窗口一样，都被纳入 Agent 的统一 UI 生命周期，不必等到第一次点击模式按钮时再临时创建。

---

### 4. UI 配置结构已预留模式面板选项
已更新：

- `aibot/Scripts/Config/AiBotConfig.cs`

新增配置结构默认值：

- `showModePanel`

当前阶段没有强制修改用户本地 `config.json`，因此旧配置也能直接兼容运行；若缺失这些字段，将自动使用默认值。

这避免了不必要的配置迁移风险，同时为后续面板增强保留了稳定入口。

## 遇到的问题

### 问题 1：模式切换逻辑不能散落在 UI 中
如果由面板直接决定如何切换模式、何时强制切换，会导致：

- 业务逻辑重复
- 其他未来入口难以复用
- 模式切换规则可能在不同 UI 中不一致

### 解决方案
继续坚持：

- `AgentCore` 负责模式切换状态机
- `AgentModePanel` 只负责触发请求、展示确认、反馈状态

---

### 问题 2：必须兼容旧配置
如果本阶段直接强制要求用户更新 `config.json`，会增加使用成本，也可能误伤已有本地配置。

### 解决方案
在 `AiBotConfig` 中先加入默认字段：

- 配置存在则读取
- 配置缺失则使用默认值

这样既能兼容旧配置，又能支持未来扩展。

---

### 问题 3：不能修改 `sts2/`
模式切换 UI 只能通过 `aibot/` 自己创建，不能依赖改动游戏侧 UI 结构。

### 解决方案
继续采用：

- `CanvasLayer`
- 纯代码构建控件树
- 挂到 Godot 根节点

从而保持 Mod 自包含实现。

---

这样可以让模式操作更接近真正可用的游戏内交互体验。
